# Review Intake

## 概述

在真正审代码之前，先确认“我有没有理解错这次变更目标”。很多 review 失败不是因为审查能力不足，而是从一开始就审偏了：以为目标是重构，实际是修复线上 bug；以为目标是补日志，实际是让某条行为上线后可追踪。

本技能通过一次只问一个问题、暴露假设并标明猜测、持续更新置信度，产出后续 `code-review` 和 `trace-review` 可以使用、且不依赖 diff 也能理解变更意图的 Review Brief。如果变更目标、预期行为、不变行为已经清楚，则不执行逐轮澄清，直接进入 Review Brief 确认闭环。

## 何时不用

- **已经有确认过的 Review Brief**，且包含变更目标、预期行为、不变行为；此时后续审查直接消费这份 Review Brief，不再调用本技能。
- 用户明确要求直接审代码，且请求中已经提供并确认了 Review Brief。

## 加载约束

此技能需要一个处于实时交互中、会响应的用户。**不要在非交互式上下文中调用**，例如 CI 流水线、定时运行、`/loop` 或 autonomous-loop。如果你处在这些环境中，而需求又信息不足，请将其标记为需要用户处理的阻塞项，而不是猜测。

## 边界

- **只**澄清 Review Brief，**不**审代码。
- **不生成**代码审查结论。
- **不判断**日志和埋点是否充分。
- **不扩展**成完整的需求分析或规格制定。
- **不收集**与“变更目标是否理解偏移”无关的信息，包括业务路径、审查维度或关键路径等后续审查材料。
- **不收集**代码实现风险、测试或验证缺口；这些属于 `code-review`。
- **不收集**日志、埋点、可追踪性或 trace 覆盖材料；这些属于 `trace-review`。

## 原则

- **暴露假设，而不是默默补全。**
- **把猜测标成猜测，让用户确认或纠正。**

## 流程

### 快速闭环：目标已清楚时直接确认 Review Brief

如果用户已经明确给出变更目标、预期行为、不变行为，且没有明显矛盾、缺口或理解偏移迹象，说明置信度已达到约 95%，**不需要**执行步骤 1-3 的逐轮澄清。直接进入步骤 4，输出待确认的 Review Brief，并按步骤 5 等待用户明确确认。

### 步骤 1：提出初始假设和置信度

**必须**先用一句话写下你认为这次变更想达成的目标，并给出 0-100% 的置信度。

格式：

```text
假设：这次变更的目标是让支付失败时能被稳定追踪，而不是重写支付流程。
置信度：约 50% —— 缺少：用户真正想解决的是“失败不可见”，还是“失败处理逻辑错误”
```

低于约 70% 时，**必须**说明目标理解上还缺什么，并继续澄清；这个数字不是装饰。70% 只是低置信度警戒线，最终出口仍然是置信度达到约 95%，且变更目标、预期行为、不变行为都不再依赖猜测。

### 步骤 2：一次只问一个问题，并附上猜测

每轮**只问一个**最能判断目标是否偏移的问题。每个问题都**必须附带你的猜测**，让用户可以快速纠正错误假设。

格式：

```text
问：这次变更主要是为了改变用户可见行为，还是为了让现有行为上线后可追踪？
猜：主要是可追踪。因为你强调上线后定位失败原因，而不是修改支付成功路径。
```

**禁止**一次列出多个问题。批量问题会让用户扫读，也会掩盖你的真实假设。

### 步骤 3：理解仍依赖猜测或置信度不足时继续追问

如果变更目标、预期行为、不变行为中任一项仍依赖你的猜测，或置信度低于约 95%，继续回到步骤 2。每次只问一个问题，并更新假设、置信度和仍缺失的确认点。

**只有**变更目标、预期行为、不变行为都不再依赖猜测，且置信度达到约 95%，**才可以**进入步骤 4。

**禁止**因为“看起来差不多懂了”而提前输出 Review Brief。

```text
更新假设：这次变更的目标是让支付失败可追踪，预期行为是失败时写入可关联的错误上下文，不变行为是原有支付成功路径不变。
置信度：约 80% —— 缺少：用户是否确认“成功路径不变”属于本次不变行为
```

### 步骤 4：重述待确认 Review Brief

当变更目标、预期行为、不变行为都已有用户明示或已被用户确认，且置信度达到约 95% 时，用用户自己的话写回待确认的 Review Brief。保持短，只确认目标是否理解偏移。

**必须**包含：

- **变更目标**：这次为什么改。
- **预期行为**：改完后行为应该变成什么样。
- **不变行为**：稳定行为，不应因为这次变更而改变的行为。

格式：

```text
以下是待你确认的 Review Brief：

- 变更目标：
- 预期行为：
- 不变行为：

请明确回复：确认 / 修正：
```

Review Brief 输出约束：

- 变更目标、预期行为和不变行为**只用于**确认目标是否理解偏移，**禁止**扩展成验收标准、测试计划、完整回归范围或兼容性清单。
- PR/commit 描述**只用于**判断这三项是否足够清楚；如果不能支撑 Review Brief，继续回到步骤 2 澄清目标理解缺口。
- **禁止**改写提交信息，**禁止**扩展成 PR/commit 标题、格式、提交历史或提交规范检查。

### 步骤 5：等待明确确认

**只有**用户明确确认后，**才可以**将已确认的 Review Brief 作为后续审查输入。是否调用 `code-review`、`trace-review`，以及是否并行调用，由外层命令或主 agent 决定。

下面这些**不算确认**：

- “你看着办”
- “都可以”
- “听起来不错”
- 用户沉默

遇到这些回答时，继续追问或给出两个具体选项让用户选择。用户修正后，吸收修正并重新输出待确认的 Review Brief。

用户明确确认后，**必须**复述一次确认块，作为后续审查唯一可消费的**已确认 Review Brief**：

```text
以下是已经确认的 Review Brief：

- 变更目标：
- 预期行为：
- 不变行为：
```

确认块只复用最后一次待确认内容，**禁止**添加代码审查、trace-review 或验证判断。

## 红旗

- 用户只说“看有没有问题”，但没有说明这次为什么改。
- 用户能说出实现方案，但说不出预期行为。
- 用户只强调 trace 或 review，但无法确认不变行为。
- 用户把“修一下”“优化一下”“补一下”当作变更目标。
- PR/commit 描述只写“优化”“修复问题”或“调整逻辑”，无法支撑变更目标或预期行为。

## 反模式

- 把“全面看一下”当作变更目标。
- 把“按标准 review”当作变更目标。
- 把“看有没有问题”当作变更目标。
- 把“保证代码质量”当作变更目标。
- 把“看看架构是否合理”当作变更目标。
- 在置信度低于约 95% 时输出 Review Brief。
- 在 Review Brief 阶段判断代码实现风险、日志埋点充分性或 trace 覆盖是否合理。
- 违反 Review Brief 输出约束，把确认阶段扩展成后续审查材料收集、验收/测试计划或 PR/commit 规范检查。

## 验证

- 已确认对变更目标的理解没有偏移。
- 已明确预期行为和不变行为。
- Review Brief 不依赖 diff 也能说明这次为什么改、改完后行为应该是什么、不应改变什么。
- 每轮澄清都是一个问题，并附带当前猜测。
- 低置信度时没有进入代码审查。
- 目标已经清楚时，未执行逐轮澄清，而是直接输出待确认 Review Brief 并等待确认。
- 已遵守 Review Brief 输出约束，未扩展成后续审查材料、验收/测试计划、PR/commit 规范检查、代码实现风险、验证缺口或日志充分性判断。
- 用户已明确确认 Review Brief，并已输出**已确认 Review Brief**确认块。
