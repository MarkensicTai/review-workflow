# Code Review

## 概述

带质量门的多维 `code-review`。当本轮进入合并前代码质量审查，且已有已确认 Review Brief 时使用。每个合并前变更都必须审查，没有例外；审查覆盖五个维度：正确性、可读性、架构、安全和性能。

## 何时不用

- 没有已确认的 Review Brief：先使用 `review-intake` 完成确认闭环。
- 用户只要求实现修改，而不是审查。
- 当前是在讨论实现前方案、需求风险或质量风险，而不是合并前代码审查。

## 批准标准

**批准标准：** 变更不需要完美；如果它满足已确认 Review Brief、没有降低整体代码健康度、遵循项目约定、没有五维审查 `Critical` 或 `Important` 问题、没有补充质量门 `[阻断]`，并且关键验证证据足够，可以批准。不要因为一个变更和你自己的写法不完全一样就阻止它。

## 审查输入

- **接受已确认的 Review Brief 作为意图来源**；不要在 `code-review` 中重新访谈、重新推断或改写变更目标。
- Review Brief 必须能提供：变更目标、预期行为、不变行为。
- 如果 Review Brief 缺失或仍不清楚，停止审查并回到 `review-intake`。

## 审查前检查

- 确认变更是一个自包含修改：只解决一件事，相关测试在同一变更中，提交后系统仍可用。
- 先识别变更规模。约 100 行适合一次审查；约 300 行只有在单一逻辑变更时可以接受；约 1000 行通常应要求拆分。
- 如果同时包含重构和行为变更，检查二者是否应该拆开。小型清理可以接受，但不能掩盖行为变化。
- 如果是大型删除或自动化重构，不逐行追求人工风格审查；但仍必须检查它是否符合 Review Brief、变更范围和生成方式是否一致、关键风险是否被覆盖，以及验证证据是否足以支撑结论。

## 审查流程

### 步骤 1：固定审查基准

先把已确认的 Review Brief 当作本次审查契约，只用它判断实现是否审偏：

- 变更目标：这次为什么改。
- 预期行为：改完后行为应该变成什么样。
- 不变行为：哪些稳定行为不应改变。

### 步骤 2：先审查测试覆盖

测试是代码审查证据，不是意图来源。意图**只**来自已确认的 Review Brief。

先看测试是否能证明变更符合 Review Brief：

- 是否有测试覆盖 Review Brief 中的预期行为和不变行为？
- 测试验证的是用户可见行为或业务契约，而不是实现细节？
- 是否覆盖边界情况、错误路径和回归风险？
- 如果代码回归，这些测试是否能失败？
- 测试名称是否能表达场景和预期结果？

bug 修复没有回归测试时，默认写入补充质量门；风险高时标为 `[阻断]`；没有足够替代验证证据时标为 `[待确认]`；只有风险低且已有足够替代验证证据时，才标为 `[建议]`。

### 步骤 3：按五维审查实现

对每个修改过的文件，按五个维度走读代码。每个发现都必须能指向具体代码、行为影响、审查维度和严重级别。

#### 1. 正确性

代码是否做到了它声称要做的事？

- 是否符合已确认的 Review Brief？
- 是否处理了边界情况，例如 null、空输入、边界值和异常状态？
- 是否处理了错误路径，而不只是正常路径？
- 是否存在差一错误、竞态条件、生命周期问题或状态不一致？
- 步骤 2 的测试证据是否足以支持正确性判断？

#### 2. 可读性与简洁性

其他工程师或智能体能否在不需要作者解释的情况下理解这段代码？

- 命名是否清晰，并与项目约定一致？
- 控制流是否直接，是否避免过深嵌套、嵌套三元表达式或过深回调？
- 相关代码是否放在一起，模块组织是否有逻辑？
- 是否存在过度巧妙、难以维护的写法？
- **能否用更少代码完成？** 100 行能解决却写了 1000 行，就是失败。
- **抽象是否配得上它引入的复杂度？** 不要在第三个真实用例出现前过早泛化。

#### 3. 架构

这个变更是否符合系统设计？

- 它是遵循现有模式，还是引入了新模式？如果是新模式，理由是否充分？
- 是否保持了清晰的模块边界？
- 依赖方向是否正确，是否避免循环依赖？
- 抽象层次是否合适，不过度工程化，也不过度耦合？
- 是否存在应该抽取共享的重复代码？

#### 4. 安全

这个变更是否引入漏洞？

- 外部输入是否在系统边界处验证和清理，例如 API、CLI、外部路由、深链、文件、配置或消息？
- 密钥是否没有进入代码、日志和版本控制？
- 需要认证或授权的地方是否已经检查？
- 查询、命令或路径构造是否使用参数化或结构化 API，而不是字符串拼接？
- 如果存在 HTML、Web 容器或富文本渲染，输出是否做了编码或转义？
- 来自 API、日志、用户内容、配置文件等外部来源的数据是否被视为不可信？
- 组件边界和跨进程入口是否安全：对外暴露组件、外部路由、深链、延迟授权对象、文件或内容共享入口是否限制调用方和数据范围？
- 敏感数据是否避免进入明文存储、本地存储组件、日志、剪贴板、截图、崩溃上报和不受保护的缓存？
- 网络与 Web 容器配置是否避免明文流量、错误证书信任、调试代理配置泄漏或不必要的脚本桥接接口暴露？

#### 5. 性能

这个变更是否引入性能问题？

- 是否存在 N+1 查询、重复远程调用或重复重计算？
- 是否存在无界循环、无限递归或不受约束的数据读取？
- 是否存在本应异步、移出主线程或移出关键路径的同步操作？
- 列表、批处理或游标读取是否缺少分页、limit 或背压？
- 热路径上是否创建了大对象？
- 主线程是否存在磁盘 IO、网络请求、JSON 解析、图片解码、数据库查询或长事务，可能导致卡顿、掉帧或无响应？
- 生命周期相关任务是否正确取消，避免页面、视图模型或控制器泄漏、重复订阅数据流、异步任务悬挂或后台任务失控？
- UI 渲染、列表、图片加载、动画和启动路径是否避免不必要刷新、重绘、重组、全量刷新、过度绘制、大图常驻内存或冷启动关键路径膨胀？

### 步骤 4：检查补充质量门

补充质量门包括：

- 测试覆盖：来自步骤 2 的测试审查结果。
- 依赖纪律：在本步骤检查新增依赖、体积、维护、漏洞和许可证。
- 死代码：在本步骤检查不可达代码、旧兼容垫片和“以后再删”的残留。

#### 依赖纪律

添加任何依赖之前，必须检查：

1. 现有技术栈能否解决这个问题。
2. 依赖体积影响，包括 APK/AAB、镜像、二进制或运行时体积。
3. 维护状态是否可接受。
4. 是否存在已知漏洞。
5. 许可证是否与项目兼容。

**规则：** 优先使用标准库和已有工具，而不是新增依赖。每个依赖都是负担。

#### 死代码

- 重构或替换后，检查是否留下不可达代码、旧兼容垫片、无作用变量、废弃常量或“以后再删”的残留。
- 明确列出疑似不再使用的代码。
- 不要静默删除不确定的内容；不确定时，先询问或标记为待确认。

### 步骤 5：核实验证证据

检查作者或当前环境提供的验证：

- 运行了哪些测试或构建？
- 是否有手动验证说明？
- UI 变更是否有截图或前后对比？
- 风险路径是否有针对性验证？

没有验证证据时，不要用“应该没问题”补全。

已核实的证据写入 `Verification Evidence`；缺失证据按验证缺口分类规则写入 `Supplemental Quality Gates` 或 `Verification Gaps`。

### 步骤 6：输出分级结论

发现优先于总结。先列五维审查发现，再列建议和补充质量门状态，保留覆盖情况，再列验证缺口和结论。

## 审查纪律

### 处理分歧

解决审查争议时，按以下优先级判断：

1. **技术事实和数据**优先于观点和偏好。
2. **风格指南**是风格问题的绝对权威。
3. **软件设计**必须基于工程原则评估，而不是个人偏好。
4. **代码库一致性**可以接受，前提是不降低整体健康度。

**不要接受“我以后再清理”。** 除非是真正的紧急情况，否则要求提交前完成清理；如果周边问题无法在这次变更中解决，要求创建后续 bug 或 issue。

### 审查中的诚实

- **不要走形式批准。** 没有审查证据的 “LGTM” 没有帮助。
- **不要淡化真实问题。** 如果一个 bug 会进入生产环境，不要把它说成小问题。
- **尽可能量化问题。** 能说明影响范围、耗时、数量级或触发条件时，不要只写“可能会慢”。
- **对明确有问题的方案提出反对。** 如果实现有问题，直接说明，并给出替代方向。
- **评论代码，不评论人。** 把针对人的批评改写成聚焦代码本身的问题。

## 输出

使用以下结构：

```markdown
## Review Basis

- 已确认 Review Brief：
  - 变更目标：
  - 预期行为：
  - 不变行为：
- 审查范围：
- 五维覆盖：正确性、可读性与简洁性、架构、安全、性能均已审查。
- 补充质量门覆盖：测试覆盖、依赖纪律、死代码均已审查。

## Findings

按严重级别分组输出，组内按影响顺序排序。五维审查发现必须同时包含严重级别和五维审查标签。

### Critical

- [Critical][正确性] path/to/File.kt:123 - 问题、影响、建议修复方向。
- [Critical][架构] path/to/File.kt:45 - 问题、影响、建议修复方向。

### Important

- [Important][性能] path/to/File.kt:67 - 问题、影响、建议修复方向。

如果某个严重级别没有发现，可以省略该严重级别小节；不要为了凑结构输出空小节。

## Suggestions

Suggestion 和 Nit 也是 code-review 输出的一部分，**不得**因为不阻止合并而静默丢弃。五维审查建议必须包含严重级别和五维审查标签。

### Suggestion

- [Suggestion][性能] path/to/File.kt:67 - 可选改进。
- [Suggestion][安全] path/to/File.kt:89 - 可选改进。

### Nit

- [Nit][可读性与简洁性] path/to/File.kt:101 - 轻微风格或格式问题。

## Supplemental Quality Gates

`测试覆盖`、`依赖纪律`、`死代码` 三个小节**必须**都出现；每个小节先输出总状态：`[阻断]`、`[待确认]`、`[建议]` 或 `[通过]`。同一小节存在多个问题时，以最高影响状态作为总状态。

### 测试覆盖

- [阻断] 总状态：缺少关键回归测试，无法证明本次 bug 修复不会回退。
- [阻断] path/to/Test.kt:12 - 缺少关键回归测试，无法证明本次 bug 修复不会回退。
- [待确认] 未提供测试运行结果，无法判断变更是否满足 Review Brief。

### 依赖纪律

- [通过] 总状态：未新增依赖。

### 死代码

- [建议] 总状态：存在可后续清理的疑似死代码。
- [建议] path/to/File.kt:89 - 旧兼容分支疑似已无调用方，可以后续清理。

## Verification Evidence

- 已核实的测试、构建、手动验证、截图或对比证据。

## Verification Gaps

- 缺失的构建、测试、手动验证、截图或对比证据。

## Conclusion

- Request changes / Approve / Unable to determine，并说明原因。
```

五维审查标注规则：

- 五维审查下的每条 `Critical`、`Important`、`Suggestion`、`Nit` 都必须写成 `[严重级别][五维审查标签]`。
- 可用五维审查标签：`正确性`、`可读性与简洁性`、`架构`、`安全`、`性能`。
- `Critical`：阻止合并。会导致安全漏洞、数据丢失、功能损坏、明显崩溃或高风险生产事故。
- `Important`：合并前应处理。会造成可见 bug、回归风险、架构退化或维护成本明显上升。测试覆盖问题按 `Supplemental Quality Gates` 处理。
- `Suggestion`：建议改进。值得考虑，但不应阻止合并。
- `Nit`：轻微、可选，通常是局部风格或格式。

补充质量门：

- `测试覆盖`、`依赖纪律`、`死代码` 问题放入 `Supplemental Quality Gates`。
- `测试覆盖`、`依赖纪律`、`死代码` 三个小节必须都出现；每个小节先输出总状态。
- 补充质量门使用处理状态：`[阻断]`、`[待确认]`、`[建议]`、`[通过]`。
- 同一小节存在多个问题时，以最高影响状态作为总状态：`[阻断]` > `[待确认]` > `[建议]` > `[通过]`。
- 补充质量门中的 `[建议]` 仍保留在 `Supplemental Quality Gates`。
- 存在 `[阻断]` 时触发 `Request changes`；存在关键 `[待确认]` 且没有阻断问题时触发 `Unable to determine`。

验证缺口分类：

- 高风险或会阻止判断是否满足 Review Brief 的验证缺口，进入 `Supplemental Quality Gates`，标为 `[阻断]` 或 `[待确认]`。
- 普通缺失证据进入 `Verification Gaps`，不和 `Supplemental Quality Gates` 重复输出。

结论规则：

- 存在 `Critical`、`Important` 或补充质量门 `[阻断]`：`Request changes`。
- 没有阻断问题，但关键验证证据不足以判断是否满足 Review Brief，或补充质量门存在关键 `[待确认]`：`Unable to determine`。
- 没有阻断问题，且验证依据足够：`Approve`。

如果没有发现问题，仍然要说明审查过的范围和验证依据，禁止只写 “LGTM”。

## 红旗

- 没有实际审查证据就批准。
- 只看测试是否通过，忽略正确性、架构、安全、性能和可读性。
- bug 修复没有回归测试。
- 大 PR 因为“太大没法认真看”而被勉强批准；应要求拆分。
- 安全敏感变更没有安全重点审查。
- 新增依赖没有必要性、体积、维护、漏洞和许可证说明。
- 接受“以后再清理”；除非是真正紧急情况，否则应在合并前处理或明确创建后续 issue。
- AI 生成代码看起来合理但缺少规格、测试或边界验证。

## 验证

- 发现均带有文件和行号证据。
- 五维审查严重级别清晰，补充质量门三个小节均已输出总状态。
- 已输出本次消费的 Review Basis。
- 已在 `Review Basis` 中说明五维覆盖和补充质量门覆盖。
- 已区分必须修改项、可选建议和验证缺口。
- 已说明本次 review 依赖的验证证据。
