# Trace Review

## 概述

带质量门的 `trace-review`。目标不是让代码“多打日志”，而是让变更上线后可追踪、可调试：当关键链路出问题、走到异常分支或出现用户可见异常时，后续排查人员能从日志、埋点、trace id、业务 marker 和错误上下文中还原发生了什么。

本技能只审查线上排障信息是否足够。它独立于 `code-review`，不依赖 `code-review` 结论，也不输出代码质量审查结论。

## 何时不用

- 没有已确认的 Review Brief：先使用 `review-intake` 完成确认闭环。
- 当前是在判断代码实现是否正确、架构是否合理、测试是否充分或依赖是否合适：使用 `code-review`。
- 用户只要求实现修改，而不是审查上线后可追踪性。
- 当前没有 diff、PR、commit 或可审查的实现，只是在讨论方案；此时可以提出 trace 设计建议，但不能输出 trace-review 结论。

## 审查输入

- **接受已确认的 Review Brief 作为意图来源**；不要在 `trace-review` 中重新访谈、重新推断或改写变更目标。
- Review Brief 必须能提供：变更目标、预期行为、不变行为。
- 需要可审查的代码变更：PR、diff、commit 或本地改动。
- 如果项目已有日志、埋点、错误上报、trace id、业务 marker、采样或隐私规范，优先按现有规范判断。
- 如果关键信息缺失，标记为 `Unable to determine` 或 `Trace Gaps`；**不要**用“应该能查到”补全。

## 批准标准

**批准标准：** 关键链路和重要分支上线后能回答核心排障问题：是否发生、发生在哪一步、影响哪个业务对象、走了哪个分支、为什么失败、能否与上下游或同一次会话关联，并且没有引入隐私、噪声、成本或性能不可接受的 trace 风险。

## 审查原则

- **以排障问题为单位审查。** 每个发现都必须对应一类上线后需要回答的问题。
- **不足的信息就是质量问题。** 如果线上异常发生后只能靠猜、复现或翻代码定位，说明 trace 不足。
- **日志、埋点和 trace id 是手段，不是目标。** 目标是让业务问题可定位、可归因、可复盘。
- **不要把 debug 日志当成线上可追踪性。** 线上需要稳定、可关联、可检索、可控成本的信号。

## 审查流程

### 步骤 1：固定 Trace Basis

先把已确认的 Review Brief 当作本次 trace 审查契约，只用它判断哪些链路和分支需要可追踪：

- 变更目标：这次为什么改。
- 预期行为：改完后行为应该变成什么样。
- 不变行为：哪些稳定行为不应改变。

### 步骤 2：识别关键链路和分支

从 diff 中找出上线后最可能需要排查的路径，而不是只找已有日志调用：

- 入口：用户操作、外部回调、定时任务、消息、深链、接口响应或状态恢复点。
- 关键节点：开始、参数校验、权限判断、状态转换、外部依赖调用、缓存读写、持久化、页面 ready、提交、成功。
- 异常分支：失败、取消、超时、重试、降级、兜底、幂等冲突、重复提交、数据不一致、解析失败、权限不足、网络异常。
- 异步边界：线程、协程、队列、回调、生命周期切换、后台任务、跨进程或跨模块调用。
- 用户可见结果：成功、失败、部分成功、无响应、空态、降级展示、错误提示或静默失败。

**禁止**只因为代码里已经有日志调用就认为链路可追踪。必须判断关键链路和分支是否都能被还原。

### 步骤 3：逐点判断信息是否足够

对每个关键链路、异常点和重要分支，检查是否能回答：

- 是否能知道这个点发生过？
- 是否能知道发生在哪个步骤、哪个分支和哪个结果？
- 是否能关联到同一次请求、会话、任务、业务对象或上下游调用？
- 是否能区分用户取消、业务拒绝、系统失败、外部依赖失败、超时、重试和降级？
- 是否记录了足够的错误上下文，例如错误码、异常类型、依赖名称、响应状态、重试次数、耗时、关键状态或分支原因？
- 是否能判断用户可见行为是否 ready、失败、超时或被降级？
- 是否能在不读源码、不复现问题的情况下缩小排查范围？

如果答案是否定的，输出缺失的 trace 点或上下文，而不是泛泛建议“加日志”。

### 步骤 4：检查信号类型是否合适

根据排障问题选择合适信号，不要求每个点同时具备所有信号：

- 日志：适合定位技术路径、错误上下文、依赖响应、状态转换和分支原因。
- 埋点：适合判断业务流程、用户可见结果、漏斗节点、关键成功/失败率。
- trace id / request id / session id：适合关联一次请求、一次任务、一次会话或上下游调用。
- 业务 marker：适合标记关键业务状态，例如 ready、submitted、confirmed、failed、timeout、fallback。
- 错误上报：适合聚合异常类型、堆栈、影响范围和版本分布。

如果使用了错误的信号类型，例如只用本地 debug 日志承载业务漏斗，或只用埋点承载异常上下文，应标为问题。

### 步骤 5：检查隐私、噪声、成本和性能边界

trace 信息必须可用，也必须可控：

- 禁止输出 token、密钥、认证信息、个人敏感信息、原始用户内容或无法脱敏的业务数据。
- 高频路径、循环、列表、重试和后台任务必须考虑采样、限流、聚合或日志级别。
- 日志和埋点字段应稳定、可检索、可聚合，避免动态 key、长文本、无界 payload 和高基数字段失控。
- trace 逻辑不能显著增加主线程、关键路径、启动路径、热路径或失败路径负担。
- 错误日志不能吞掉异常，也不能因为记录失败改变原业务行为。

隐私泄露、成本失控或 trace 逻辑改变业务行为，应作为阻断问题处理。

### 步骤 6：输出分级结论

发现优先于总结。先列必须补齐的 trace 问题，再列建议，最后说明已核实证据、Trace Gaps、仍无法追踪的问题和结论。

## 最低信息集

不是每条日志或埋点都必须包含以下全部字段，但每个关键链路和异常分支必须能组合出足够信息：

- 稳定的事件名、日志 tag 或 marker 名称。
- 结果状态：start、success、fail、cancel、timeout、retry、fallback 等。
- 关联标识：request id、trace id、session id、task id、业务对象 id 或上游调用 id。
- 分支原因：校验失败、权限不足、外部依赖失败、状态不一致、用户取消、幂等命中等。
- 错误上下文：错误码、异常类型、依赖名称、响应状态、重试次数、耗时或关键状态。
- 用户可见状态：页面是否 ready、操作是否提交、提示是否展示、是否降级或静默失败。
- 版本和实验上下文：版本号、配置、feature flag、实验组或灰度状态；只在与排障相关时需要。

## 输出

使用以下结构：

```markdown
## Trace Basis

- 已确认 Review Brief：
  - 变更目标：
  - 预期行为：
  - 不变行为：
- trace-review 范围：

## Findings

- [Critical] path/to/File.kt:123 - 缺失或错误的 trace 点；说明线上会无法回答的问题、影响和建议补齐方向。
- [Important] path/to/File.kt:45 - 上下文不足或关键分支不可关联；说明排障影响和建议补齐方向。

## Suggestions

- [Suggestion] path/to/File.kt:67 - 可选的可追踪性改进。
- [Nit] path/to/File.kt:89 - 命名、字段一致性或局部格式建议。

## Trace Evidence

- 已核实的日志、埋点、trace id、业务 marker、错误上报、配置或项目规范证据。

## Trace Gaps

- 缺失或无法判断的 trace 证据、上下文或仍无法定位的问题。

## Trace Coverage

- 已覆盖的关键链路/分支：
- 仍无法追踪的 bug 或故障模式：

## Privacy / Noise / Cost

- 隐私、安全、噪声、成本或性能边界结论。

## Conclusion

- Request changes / Approve / Unable to determine，并说明原因。
```

严重级别：

- `Critical`：阻止上线。关键用户可见失败、核心业务链路、数据状态异常或隐私泄露上线后无法定位，或 trace 本身会造成隐私/成本/性能严重风险。
- `Important`：上线前应处理。重要分支、异常点或外部依赖缺少足够上下文，会明显增加排障成本。
- `Suggestion`：建议改进。能提高排障效率，但不影响核心问题定位。
- `Nit`：轻微、可选，通常是字段命名、格式或局部一致性。

结论规则：

- 存在 `Critical` 或 `Important`：`Request changes`。
- 缺少 Review Brief、diff，或缺少足以判断关键链路的必要上下文，导致无法判断关键链路是否可追踪：`Unable to determine`。
- 关键链路和重要分支可追踪，且隐私、噪声、成本和性能边界可接受：`Approve`。

Trace Gaps 分类：

- 高风险或会阻止判断关键链路是否可追踪的缺口，进入 `Findings`，按影响标为 `Critical` 或 `Important`。
- 普通缺失证据、上下文不足或无法判断项进入 `Trace Gaps`，不和 `Findings` 重复输出。

## 红旗

- AI 生成代码只实现功能，没有留下任何可排障信号。
- 只在成功路径打点，失败、取消、超时、重试、降级分支不可见。
- 只有“发生错误”日志，没有错误码、分支原因、关联 id 或业务对象上下文。
- 只能靠复现、读源码或人工猜测判断用户走到了哪个分支。
- 日志和埋点无法关联同一次请求、会话、任务或业务对象。
- debug 日志被当成线上 trace 质量门。
- 高频路径无采样、限流或聚合，导致噪声、成本或性能风险。
- 日志、埋点、错误上报中包含 token、密钥、个人敏感信息、原始用户内容或无法脱敏的业务数据。
- trace 逻辑改变业务行为，或记录失败影响原流程。

## 反模式

- 泛泛要求“多加点日志”，但说不清要回答哪个线上排障问题。
- 把 trace-review 做成代码质量审查，评论命名、抽象或测试覆盖。
- 只检查有没有日志调用，不检查关键链路和异常分支是否能被还原。
- 把 PR/commit 描述当作 trace 证据；trace 证据必须来自代码、配置或现有规范。
- 用“应该能从服务端查到”“应该有统一上报”替代实际证据。
- 为了排障输出过量、敏感或不可控的信息。

## 验证

- 已消费已确认的 Review Brief 作为 Trace Basis。
- 已识别关键链路、异常点和重要分支。
- 每个发现都对应一类上线后需要回答的排障问题。
- 每个发现都有文件和行号证据，或明确说明缺少可判断证据。
- 已区分必须补齐项、可选建议和无法判断项。
- 已输出 `Trace Evidence` 和 `Trace Gaps`。
- 已说明仍无法追踪的 bug 或故障模式。
- 已检查隐私、安全、噪声、成本和性能边界。
- 没有输出 `code-review` 结论。
